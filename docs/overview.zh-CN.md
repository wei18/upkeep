# upkeep 工作原理

## 问题背景

仓库会持续积累偏差。函数经过重构，但对应的文档注释仍停留在旧版本；规范描述的行为在三个迭代前就已修改；翻译版 README 落后于英文原版一个版本；图标文件已被替换，旧文件却仍留在仓库中。这些问题都不会导致 CI 失败 — 它们只是悄无声息地侵蚀所有书面内容的可信度。

定期的人工审查能发现其中一部分，但这要求审查者同时在脑中维护所有产物的完整上下文。而这恰恰是一组各司其职的 AI 审查器所擅长的场景。

## 流水线：扇出 → 汇总 → 报告

upkeep 以可复用的 `workflow_call` workflow 形式运行，由四个确定性阶段组成。

**1. 发现（Discovery）**

发现步骤遍历仓库，生成结构化的文件清单：源文件、文档、规范、架构图、图片、图标、语言文件，以及约定文件（CLAUDE.md、`.claude/skills`、workflow 定义）。该清单作为所有审查器的共享输入。

**2. 并行审查（扇出）**

每个启用的审查器作为独立的 matrix job 运行。各 job 并行执行，并具备容错能力 — 某个审查器失败不会阻塞其他审查器。每个审查器接收文件清单及其专属职责，并将发现结果写入结构化输出。

**3. 汇总（Synthesis）**

单一汇总步骤读取所有审查器的发现结果，识别跨审查器的共性主题 — 例如，某个目录下集中出现约定偏差，或多个审查器出于不同原因独立标记了同一文件。汇总步骤在各审查器详情的基础上生成执行摘要。

**4. 报告（Report）**

确定性报告步骤将所有内容渲染为独立 HTML 文件（无外部依赖），并以 upsert 方式更新 GitHub 跟踪 issue。同一 issue 在多次运行间复用，保持 issue 跟踪器的整洁。

## 审查器

| 审查器 | 检查内容 |
|---|---|
| `docs_staleness` | 脱离所描述代码的文档；与基准语言版本不同步的多语言 README 及翻译文档 |
| `code_hygiene` | 死代码路径、未使用的导出、永久残留的注释代码块 |
| `spec_flow` | 内容不再符合实际实现的规范、架构图和流程图 |
| `visual_icon` | 过时、不匹配或与当前 UI / 品牌不一致的图片和图标 |
| `duplicate_orphan` | 重复文件（不同路径或文件名下内容相同或高度相似）及已提交但从未被引用的孤立资源 |
| `convention` | 偏离仓库自身声明约定的情况 — CLAUDE.md 规则、`.claude/skills` 模式、workflow 标准 |
| `i18n` | 语言文件间的一致性（默认禁用；通过 `.claude/audit.yml` 启用） |

## 不预设权威来源

当两个产物存在分歧 — 例如规范描述了行为 X，而代码实现了行为 Y — upkeep 不会自动判定其中一个已过时。任何一方都可能是陈旧的那个。upkeep 会以支撑证据的方式呈现这一分歧：每个文件的 git 最近修改时间、在其他文件中发现的交叉引用，以及任何显式的版本标记。由人来决定如何修复。

## 仅报告，从不编辑

upkeep 对仓库内容没有写权限。它只读取文件，并创建或更新单个 GitHub issue。它不会修改、重命名或删除仓库中的任何文件。

## 输出

**HTML 报告** — 每次运行时作为 `report-html` workflow artifact 上传。独立文件，无需服务器即可在本地打开。包含执行摘要、各审查器发现结果，以及每条发现所引用的证据。

**GitHub 跟踪 issue** — 首次运行时创建，后续每次运行时更新（upsert）。默认标记为 `audit`。提供对仓库当前健康状况的持久化、可链接记录，不会在 issue 跟踪器中产生重复条目。
